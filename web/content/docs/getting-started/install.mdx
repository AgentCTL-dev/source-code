title: Install on kind

This page gets the agentctl control plane running on a local
[kind](https://kind.sigs.k8s.io/) cluster. The same chart installs unchanged on
any conformant Kubernetes cluster — see the
[Helm chart reference](/docs/reference/helm-values) and the
[operations runbook](/docs/operations) for production notes.

## Prerequisites

| Requirement | Required? | Why |
|---|---|---|
| **kind** + **kubectl** + **Helm 3** | always | local cluster + install tooling |
| **cert-manager** (>= 1.13) | **always** | issues every serving/mTLS cert and injects the `caBundle`s |
| Aggregation layer enabled | always | the APIServer registers an `APIService` (standard on kind) |
| **Postgres** | bundled | the A2A gateway + ModelGateway durable store — **bundled by the chart** by default (set `postgres.mode=external` for a managed DB) |
| **KEDA** (>= 2.x) | *optional* | **only** for claim-mode **autoscaling** (`scaler.enabled` — elastic-from-zero AgentFleets). The control plane installs and runs fully without it. |
| A NetworkPolicy CNI (Calico/Cilium) | *optional* | **only** if you set `networkPolicies.enabled=true`; kindnet ignores NetworkPolicies. See [NetworkPolicies](/docs/guides/security/netpol). |

**cert-manager is the only hard external prerequisite.** Postgres is bundled. KEDA
and a policy CNI are needed *only* for the opt-in features that use them (see
[what gets installed](#what-gets-installed), below).

## 1. Create the cluster

```console
kind create cluster --name agentctl
```

## 2. Install cert-manager

cert-manager mints the agentctl CA and every serving/mTLS leaf, and rotates them
automatically. Install it and wait for it to be ready:

```console
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml
kubectl -n cert-manager rollout status deploy/cert-manager-webhook
```

## 3. Pre-create the namespace

The control-plane namespace needs a relaxed PodSecurity level (the node-agent
uses `hostPath` + `hostPID`), and Helm cannot reliably own the namespace it
installs into, so create it first:

```console
kubectl create namespace agentctl-system
kubectl label namespace agentctl-system \
  pod-security.kubernetes.io/enforce=privileged \
  pod-security.kubernetes.io/warn=privileged
```

## 4. Install the chart

From a checkout of the repository (kind-loaded local images):

```console
helm install agentctl ./charts/agentctl -n agentctl-system
```

Or pull the published chart and images from GHCR (no local image loading):

```console
helm install agentctl oci://ghcr.io/agentctl-dev/charts/agentctl \
  -n agentctl-system --version 0.1.0 \
  --set image.registry=ghcr.io/agentctl-dev --set image.tag=0.1.0
```

## 5. Verify

```console
kubectl -n agentctl-system get pods                          # components Running
kubectl -n agentctl-system get certificate                   # all READY=True
kubectl get apiservice v1alpha1.management.agents.x-k8s.io    # AVAILABLE=True
```

After install, `helm test agentctl -n agentctl-system` runs a connectivity probe
that checks the gateway `/healthz` and that the aggregated APIService is
registered and available.

## What gets installed

A default `helm install agentctl` brings up the **core control plane** — seven
components, no extra prerequisites beyond cert-manager:

`operator` · `node-agent` (DaemonSet) · aggregated `apiserver` · A2A `gateway` ·
`modelgateway` · `admission` webhook · bundled `postgres`.

Two planes are **opt-in** (default off), because they pull in extra dependencies or
are only needed for fleet workloads:

| Component | Enable | Adds |
|---|---|---|
| **coordination** server (`work.*` distribution) | `--set coordination.enabled=true` | the work-claim hub for claim-mode fleets |
| **scaler** (KEDA external scaler) | `--set scaler.enabled=true` | elastic-from-zero autoscaling of claim-mode AgentFleets — **requires KEDA** |

To use claim-mode autoscaling, install **KEDA** first, then enable the scaler:

```console
helm repo add kedacore https://kedacore.github.io/charts && helm repo update
helm install keda kedacore/keda --namespace keda --create-namespace

helm upgrade agentctl ./charts/agentctl -n agentctl-system \
  --reuse-values --set coordination.enabled=true --set scaler.enabled=true
```

> **CRDs on upgrade.** The chart ships the `Agent` / `AgentFleet` / `ModelPool` CRDs
> in `crds/`, which Helm installs automatically on **first install** but does **not**
> update on `helm upgrade`. When upgrading to a chart version with CRD changes,
> re-apply them: `kubectl apply -f charts/agentctl/crds/` (or `deploy/crds/`).

## What cert-manager wired

A self-signed bootstrap `Issuer` mints the **agentctl CA**; a CA `Issuer` then
mints every leaf — the APIServer serving cert (CA injected into the
`APIService.spec.caBundle`), the admission webhook serving cert, the node-agent
mTLS server cert, and the control-plane mTLS client cert. Renewal is automatic
(`renewBefore: 720h`). See [the certificate fabric](/docs/guides/security/overview).
