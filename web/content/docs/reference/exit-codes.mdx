---
title: Exit codes
description: The frozen Agent Control Contract exit-code table — what each code means and how the control plane interprets agent termination.
---

A conformant agent honors the frozen **exit-code table** so the control plane can
interpret termination uniformly. The table is a frozen DATA catalogue
(`exit-codes.table.json`, exit-codes version `1.0`); the source of truth is
`contract/schemas/exit-codes.table.json`.

| Code | Name | Meaning |
|---|---|---|
| `0` | `EXIT_OK` | success — one-shot completed / loop clean bound / **clean SIGTERM drain** |
| `1` | `EXIT_FAILURE` | generic/unspecified failure (also maps `Cancelled` and `Crashed`) |
| `2` | `EXIT_USAGE` | config/usage error — validation failed **before any side effect** |
| `3` | `EXIT_PARTIAL` | a partial result was emitted (a result-body property, not a status) |
| `4` | `EXIT_INTELLIGENCE` | intelligence endpoint unreachable / auth error after retries |
| `5` | `EXIT_SEMANTIC` | ran correctly but concluded the task cannot be done / refused |
| `6` | `EXIT_MCP` | a required MCP server failed connect/handshake or died unrecoverably |
| `7` | `EXIT_BUDGET` | hit max-steps / max-tokens / deadline / tree budget before a result |
| `124` | `EXIT_TIMEOUT` | hard wall-clock deadline tripped (mnemonic to `timeout(1)`) |
| `137` | `SIGKILL_EXIT` | `128+SIGKILL` — OOM / kubelet hard-kill (raise the memory limit); kernel-set |
| `143` | `SIGTERM_EXIT` | `128+SIGTERM` — **ungraceful** exit (drain forced past budget); distinct from a clean drain (`0`) |

## Notes

- **A clean drain exits `0`**, not `143`. `143` means the drain was forced past its
  budget and the process was signalled.
- `124`, `137`, `143` are set by the supervisor/kernel, not chosen by the binary.
- `EXIT_USAGE` (`2`) is the contractual signal that validation failed before any
  observable effect — e.g. a typo'd config key (the config-file input surface is
  intentionally closed; see [the ACC spec](/docs/contract-spec)).

The control plane surfaces termination through `Agent.status` (see the
[CRD reference](/docs/reference/crd-reference)) and metrics like
`agent_runs_total` by terminal status (see [metrics](/docs/reference/metrics)).
